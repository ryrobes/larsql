Auto-Fix: Self-Healing Cells

When SQL or Python fails, the error becomes context for an LLM that generates a fix and retries. No manual intervention. Your pipeline debugs itself. 

on_error: auto_fix
# or customize:
on_error:
  auto_fix:
    max_attempts: 3
    model: claude-sonnet-4

🗃

LLMs as SQL Functions

Call LLMs directly from any SQL client—DBeaver, psql, Tableau. Extract, classify, or run entire workflows per row. 

SELECT product_name,
  rvbbit_udf('Extract brand',
             product_name) as brand
FROM products;

One workflow, any language

SQL queries, Python transforms, JavaScript validation, LLM reasoning—each cell does what it's best at. Session-scoped temp tables flow data between languages with zero copying.

Polyglot cascade: mixed cell types 
cells:
  - name: get_customers        # SQL - no LLM
    tool: sql_data
    inputs:
      query: "SELECT * FROM customers"
      # Creates temp table: _get_customers

  - name: enrich               # Python - no LLM
    tool: python_data
    inputs:
      code: |
        df = duckdb.sql("SELECT * FROM _get_customers")
        df['segment'] = classify(df)

  - name: analyze              # LLM reasoning
    instructions: "Analyze these segments..."
    context:
      from: [get_customers, enrich]

  - name: visualize            # JavaScript
    tool: js_data
    inputs:
      code: "return createChart(data);"
SQL → Python → LLM → JavaScript

Zero-copy temp tables

Each cell output becomes a DuckDB temp table (_cell_name) that downstream cells query directly. No serialization overhead.

Deterministic cells

Skip LLM mediation when you don't need it. Direct tool invocation for SQL, Python, shell commands. 10x faster, predictable costs.

Mixed workflows

Combine deterministic data movement with LLM reasoning. Use AI where it matters, skip it where it doesn't.

Polyglot validators: validation in any language 
wards:
  - mode: retry
    max_attempts: 3
    validator:
      python: |
        # Python validation
        return {"valid": len(output) > 100, "reason": "Too short"}

  - mode: blocking
    validator:
      sql: |
        -- SQL validation
        SELECT COUNT(*) > 0 as valid FROM parse_json(output)
        WHERE required_field IS NOT NULL

Context-level observability

Every LLM call is N context chunks plus your prompt. Most frameworks show total cost. RVBBIT shows what's inside: per-chunk cost, relevance, and lineage tracking.

The Context Inspector
Hash-tracked chunks

Every piece of context has a unique hash. Track it across the entire workflow.

Cost attribution

"This system prompt is 61% of your cost but 0% relevance." Now you know what to cut.

Tiered context management

Sliding window, observation masking, loop compression. Never drop information—just compress what the LLM sees.

Selective context DSL

Control exactly what context flows between cells: context: {from: [cell_a, cell_c]}

Stop guessing what matters

See which context chunks actually influenced output. Optimize with data, not intuition.

Token budget enforcement

Set max tokens per phase. Auto-prune strategies: sliding window, summarize with cheap model, or fail with clear error.

Tool result caching

Content-addressed cache with per-tool TTL. Same SQL query 3 times? Only first hits the database.

What RVBBIT is not
Not a chatbot framework

Optimized for workflows, not conversations. No "agent personas."

Not "autonomous agents"

Explicit workflows with explicit control. Humans in the loop, not out of it.

Not DAG spaghetti

Complexity contained within phases ("separate bowls"). Your cascade stays clean.

"Keep the spaghetti in separate bowls."

Traditional frameworks put all complexity in the graph. Your clean 4-node DAG becomes 40 nodes with 100 edges. RVBBIT uses phases as complexity containers: all iteration, retries, and exploration happen INSIDE phases. Your cascade stays linear. The mess stays encapsulated.

The architecture

No central server required. Each component is independent. Scale by running more processes.

CLI

rvbbit run workflow.yaml
Cron it. CI/CD it. Script it. Each run is independent.

Studio

Visual workflow builder and execution monitor. Optional—workflows run fine without it.

SQL Server

PostgreSQL wire protocol. Connect from DBeaver, use rvbbit_udf() in queries.

All three share the same core modules. No message queues, no Redis, no coordinator. Just pip install and run. Zero infrastructure required.
