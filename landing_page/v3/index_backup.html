Not another agent framework

Most frameworks optimize for chatbots with short context. RVBBIT optimizes for real work: long-running pipelines, huge context windows, and the question everyone ignores— "what context actually mattered?"

Declarative, not code-first

Workflows are YAML files you can diff, review, and version control. The cascade IS the artifact.

Hybrid execution

Zero infrastructure

Context-level observability

Every LLM call is actually N context chunks plus your prompt. Most frameworks show you total cost. RVBBIT shows you what's inside: per-chunk cost, relevance scores, and lineage tracking.

The Context Inspector
Hash-tracked chunks

Every piece of context has a unique hash. Track it across the entire workflow.

Cost attribution

"This system prompt is 61% of your cost but 0% relevance." Now you know what to cut.

Relevance analysis

Post-hoc analysis of which context actually influenced the output, with evidence.

Context lineage

Summarized context? Track the parent chunks. Trace any output back to its source.

Stop guessing what the LLM needs

Relevance scores tell you what context mattered. Evidence explains why. Optimize with data, not vibes.

Selective context DSL

Once you see the snowball, control it: context: {from: [cell_a, cell_c]}

Waste detection

"2,075 tokens never re-used." Find the context that's costing you money but not helping.

"Magic is cool, but debugging magic is not fun."

LLMs are inherently opaque. But the orchestration around them doesn't have to be. RVBBIT puts glass walls around everything except the model itself. See what went in, what came out, what it cost, and whether it was worth it.

Polyglot, hybrid workflows

One workflow, multiple languages. SQL queries, Python transforms, JavaScript validation, LLM reasoning—each cell does what it's best at.

YAML cascade: mixed cell types 
cells:
  - name: get_customers        # SQL - no LLM
    tool: sql_data
    inputs:
      query: "SELECT * FROM customers"

  - name: enrich               # Python - no LLM
    tool: python_data
    inputs:
      code: "df['segment'] = classify(df)"

  - name: analyze              # LLM reasoning
    instructions: "Analyze these segments..."
    context:
      from: [get_customers, enrich]

  - name: report               # SQL - no LLM
    tool: sql_data
    inputs:
      query: "INSERT INTO reports..."
SQL interface: LLMs as functions 
-- Call LLMs from any SQL client
SELECT
  product_name,
  rvbbit_udf(
    'Extract the brand name',
    product_name
  ) as brand
FROM products;

-- Run full cascades per row
SELECT rvbbit_cascade_udf(
  'traits/classify.yaml',
  json_object('text', review_text)
) as classification
FROM reviews;

Works from DBeaver, psql, Jupyter—any SQL client.

What RVBBIT is not

Not "autonomous agents"

The architecture

No central server required. Each component is independent. Scale by running more processes.

CLI

rvbbit run workflow.yaml
Cron it. CI/CD it. Script it. Each run is independent.

Studio

Visual workflow builder and execution monitor. Optional—workflows run fine without it.

SQL Server

PostgreSQL wire protocol. Connect from DBeaver, use rvbbit_udf() in queries.

All three share the same core modules and write to the same ClickHouse database. No message queues, no Redis, no coordinator. Just Python imports and a shared audit log.
